### 架构层面

### 阶段 4：FPS 优化（折腾半天，教训多）
**诊断循序**：
1. 初版跑出 14.5 FPS（带 CLEAR_LOOP），比 CPU 还慢
2. 发现 HLS "cycle count" 乐观，实际 AXI+DDR 有 ~14× 延迟
3. 把 CLEAR_LOOP 从 IP 搬到 ARM：**单 slice 954→360 μs**，跳到 38 FPS
4. 把 FCLK_CLK1 从 75→150 MHz：**效果几乎为 0**，说明 IP 实际在等 DDR 而非内部逻辑
5. 改用 FCLK_CLK3 专门 150 MHz 给 pov（FCLK_CLK1 必须留 75MHz 给 rgb2dvi 的 PLL）
6. 试了 memcpy(6 byte)、memcpy(4 byte) pack 像素，**都比原始 byte-by-byte 更慢**（HLS memcpy 有开销 + II 变差）
7. 最终回到 2×2 byte 写，维持 28.59 FPS

**关键发现**（重要）：
- HLS `Latency (cycles)` 估算偏差 ~14×。每点实际约 1.15 μs。
- 瓶颈 ≠ IP 内部逻辑，而是 m_axi_gmem1 → HP1 → DDR 的真实延迟
- ARM 轮询 pov_r(POV_AP_CTRL) 每次 ~477 ns（GP0 + smc + CDC 到 150MHz + 回），不浪费
- 提高 IP 时钟对 AXI write 延迟帮助有限

### 阶段 5：点云流协议
- 设计极简协议（避开现有为 mesh 设计的 pov3d_protocol.h）：
  ```
  header (16B): magic 0x4C435050 "PPCL" | frame_id | num_points | flags | reserved
  points: num_points × 16-byte point_t
  ```
- Host：`host/pointcloud_stream.py`（旋转+色相循环立方体生成）、`host/send_axes_once.py`（调试用，只发轴）、`host/uart_logger.py`（共享 log 给 Claude + 用户）、`host/pointcloud_proto.py`（打包）
- ARM：在 `helloworld.c` 加字节级状态机 `uart_poll_frame()`：
  - 滑动 4 字节窗口搜 magic
  - 匹配后读完 16B header + N×16B point 数组
  - 完成后设 `model_n` + `Xil_DCacheFlushRange`
  - 每 slice 间插 uart_poll 防 RX FIFO（64B）溢出

## 关键 bug & 修复记录（为未来的自己）

### Bug 1：HLS export_design 有残留 Verilog
**现象**：新版 HLS 综合后，impl/ip/hdl/verilog/ 下 pov_project.v 是**老时间戳**，仍指向旧 Pipeline 模块（含已删除的 CLEAR_LOOP）
**根因**：HLS incremental 未完整重生成
**修法**：`rm -rf hls_proj/proj_pov/solution1/impl` 后重跑 HLS
**文件**：`hls_proj/proj_pov/solution1/impl/ip/hdl/verilog/pov_project.v` 时间戳要跟 csynth 同时

### Bug 2：BD 里 IP 被 "locked"
**现象**：Vivado 跑 validate_bd_design 报 `This command cannot be run, as the BD-design is locked`
**根因**：新 IP revision 不同，Vivado 自动把 BD 里老 IP 实例 lock 住等手动升级
**修法**：`upgrade_ip [get_ips -filter "IS_LOCKED==1"]` 加到 tcl 开头

### Bug 3：rgb2dvi PLL VCO 越界
**现象**：FCLK_CLK1 改 150 MHz 后，rgb2dvi 内部 PLL 报 `VCO 2250 MHz 超出 800–1866 MHz`
**根因**：rgb2dvi 用 FCLK_CLK1 驱 PLL，CLKFBOUT_MULT=15 × 150MHz = 2250MHz 越界
**修法**：FCLK_CLK1 保留 75 MHz，专开 FCLK_CLK3=150 MHz 给 pov 路径；axi_smc 加 2-clock 做 CDC

### Bug 4：axi_smc Bus Interface CLK_DOMAIN 不匹配
**现象**：`CLK_DOMAIN does not match between /pov_project_0/s_axi_control (FCLK_CLK3) and /axi_smc/M03_AXI (FCLK_CLK1)`
**修法**：`set_property CONFIG.ASSOCIATED_BUSIF {M03_AXI} [get_bd_pins axi_smc/aclk1]` 告诉 smc M03 走 aclk1 (FCLK_CLK3)

### Bug 5：ARM 读错 UART（console 是 UART0 不是 UART1）
**现象**：ARM 加 UART RX 代码后整板 hang，串口不再吐
**根因**：我用 UART1@0xE0001000 但 BSP 的 STDOUT_BASEADDR=0xE0000000 (UART0)；UART1 的 SR 寄存器读不合理值让 while 循环永远进去
**修法**：改 UART0 + while 加 `guard=256` 上限
**教训**：`grep STDOUT_BASEADDR` in BSP include 确认

### Bug 6：xil_printf 不支持 %lx / %08lx
**现象**：`printf("0x%08lx", (unsigned long)MODEL_ADDR)` 打出栈垃圾 `0x13B11000000`
**修法**：强转 `(unsigned)` 用 %x

### Bug 7：XTime_GetTime() 返回 0
**现象**：FPS 计算全 0（ticks 差 0）
**根因**：新 BSP 的 xiltimer 配置变了
**修法**：直接读 A9 Global Timer 寄存器 0xF8F00200/0x204（3ns/tick，333MHz）作为备用时基
```c
#define GT_CNT_LO 0xF8F00200U
#define GT_CTRL   0xF8F00208U
```

### Bug 8：Windows 串口缓冲残留导致错位
**现象**：第一次 stream 收到 `N=123`（不是 host 发的 279）
**根因**：之前 `send_axes_once.py` 发了 123 点，Windows UART 缓冲没空，新 stream 启动时先吃到旧 bytes
**修法**：测试前先 `rst -system` 板子 + 短时 `ser.read()` 清 Windows 缓冲

### Bug 9：UART FIFO（64 byte）在 ARM 长操作时溢出
**现象**：连续 stream 时点数据部分丢失（HDMI 上只剩散点）
**根因**：ARM memset（2.76 MB ~5-8 ms）+ `Xil_DCacheFlushRange` 期间 UART 进 ~90 byte 超过 FIFO
**修法**：在 72-slice 循环中每 slice 间插 `uart_poll_frame()` 保持 FIFO 下水

### Bug 10：Vivado bitstream incremental synth 不 reset
**现象**：BD lock 被解后 synth/impl 又跑了一次才对上
**教训**：BD 变化大的时候用 `reset_run synth_1 && reset_run impl_1` 强制从头

### Bug 11：VDMA `EnableCircularBuf=0` + `XAxiVdma_StartParking` 切换让 VDMA 完全卡死
**现象**：HDMI 黑屏，VDMA RS 位为 0
**修法**：初始化时就用 park 模式（`EnableCircularBuf=0`）+ StartParking 一次设 idx=0，后续只用直写 PARK_PTR_REG 的方式切换（不调 API 避免 stop/restart）

### Bug 12：`XAxiVdma_DmaSetup` 没有 `FrameStoreStartIndex` 成员
**现象**：编译报错 `did you mean 'FrameStoreStartAddr'?`
**修法**：删掉这行，HLS driver 不需要它

### 未完成 bug：VDMA park_ptr 写不生效
**现象**：`parkW` 读回永远 0，`curR` 固定 1，ARM 切 buf 写但 VDMA 不跟
**怀疑**：PARK_PTR 位定义不对 或 VDMA 在 circular_park=0 模式下不响应运行时改 PARK_PTR
**下一步**：加 readback 确认写是否 latch；或换 `XAxiVdma_ChannelSetFrmStore()` 驱动 API

## 性能分析（写给未来继续优化用）

### 当前 35 ms/帧 拆解
| 段 | 时间 | 能不能压 |
|---|---|---|
| ARM memset 2.76 MB fb | ~5-8 ms | 能：换 DMA 硬件清屏、或写 IP 做 |
| ARM draw borders | ~0.5 ms | 不管 |
| Xil_DCacheFlushRange(fb) | ~1-2 ms | 能：改 fb 为 uncached |
| 72 × pov_render_slice 调用 | ~26 ms | 能：并行 IP、降 AXI 延迟 |
| └ 9× AXI-Lite 写参数 | 4 μs / slice = 288 μs 合计 | 不大 |
| └ ap_start 触发 + 轮询 ap_done | 350 μs / slice = 25 ms 合计 | 主要 |
| └ IP 真实计算 | ~25 μs / slice = 1.8 ms 合计 | 已经很小 |

**结论**：单路 IP 架构下，ARM memset + IP AXI 延迟各占一半。

### LED 板最终需求分析
- 720 slice × 30 Hz = **21600 slice/sec**
- 每 slice 预算 = **46 μs**
- 当前：**355 μs**，差 **8× 性能**

### 路径选择（今天讨论结果）
| 优化 | 预期收益 | 工作量 |
|---|---|---|
| 4 路并行 IP | 4× slice throughput | 8-12 h |
| m_axi 64-bit 数据宽度 | 2× write throughput | 4 h |
| HLS dataflow + 内部 buffer | 潜在大 | 8 h |
| 中断替代轮询 | ≈0（IP 真的忙 350μs）| 4 h |

### 架构选择（今天确定）
**架构 C 混合 ring buffer**：
- DDR 里 N 个 slice slot（ring buffer，N=8 约 720KB）
- HLS IP 以"当前空 slot"为目标渲染
- LED driver IP 从 ring buffer 按 hall sensor 角度取 slice
- 抖动由 buffer 吸收，死线放宽为"平均 46 μs"

## 今天的架构讨论结果

### "通用 POV3D 显示器"的数据层
- 确定走 **Host-heavy**：PC 端 mesh → 点云转换、格式 loader 都放 PC；板子只认点云
- 原因：板子 HW 一次定型就不变；Host 加格式只改 Python，无需重综合；PC GPU 便宜
- Host 栈：Open3D/Blender/three.js 都能生成点云

### 最终 LED 板参数（从用户确认）
- 720 slice / 360° 旋转
- 30 Hz 体刷新
- 每 slice 160 × 180 pixel
- 15-30 路并行 RGB 输出（ICND1069 驱动）
- HDMI 预览 1920 × 1080（刚好 160×180 × 72）

### 时间预算（体刷新）
- 每 slice 46 μs 严格
- 框架 C 混合 ring buffer 允许平均 46 μs，不需要硬死线
