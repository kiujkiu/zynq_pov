## 主线产出

### 1. Host 端 anime 流（早段）

新建 `host/glb_to_points.py`：用 `pygltflib` 解析 glb，按三角形面积加权采样点云。带：
- 纹理 UV 采样（读 base color 贴图）
- 4 种颜色模式：`keep` / `uniform` / `height` / `random`
- gamma + brighten 调色（保色调归一化避免饱和）

新建 `host/anime_stream.py`：循环动画发送（绕 Y 旋转 + 呼吸 + 浮动）。

新建 `host/pointcloud_proto.py` 增强：**5 字节压缩格式**（int8 x,y,z + RGB565），减少 UART 带宽 3.2×。

### 2. 板子端 PPCL 协议解码（早段）

`helloworld.c` UART RX 状态机扩展支持压缩点：每点根据 flags bit 0 自动选 5B 或 16B 解析。RGB565 解包还原回 RGB888。

UART RX FIFO 64 byte，每次 CPU 渲染长 panel 时容易溢出。解决：在每 panel 渲染循环中**每 128 点 poll UART 一次**。

### 3. 双 panel 布局（中段）

把 12×6 grid 改成两个大 panel 并列：
- 左 640×720：完整 3D 投影
- 右 640×720：z 切片

加入 `cpu_render_panel()` 灵活渲染函数（参数：panel 起点、宽高、slice 模式、缩放）。

### 4. HDMI 采集卡接入（晚段）

用户加了 USB3 HDMI 采集卡（C1-1 USB3 Video，VID 345F:2130）。

OpenCV DSHOW/MSMF 后端**看不到**这个设备（看到的是笔记本相机）。改用 ffmpeg 的 dshow 输入直连，**1280×720 @60 fps YUY2 抓帧成功**。

工具：`tools/capture_ffmpeg.py`（imageio 自带 ffmpeg）。

### 5. 36-slice 网格（最后）

用户要求最终布局：
- 左半：完整 3D 投影 10s/圈旋转
- 右半：6×6 = 36 切片网格，每 10° 一个角度，**全部静态同时显示**

`cpu_render_panel` 复用得很好，36 切片每格 ~106×120 px。

### 6. Host GUI 软件（晚 8 点）

`host/pov_gui.py`：tkinter 界面，拖入 glb 自动采样发送。功能：
- 文件对话框选 .glb/.gltf/.obj/.ply
- 调参（点数、缩放、颜色、亮度、gamma）
- 串口自动检测下拉
- 静态发 / 动画流 / 停止
- 实时日志窗口（板子 UART 回传 + tx 状态）

## 关键 bug 修复

### B1: COM7 → COM9 漂移
板子 reboot 后 USB 重新枚举，串口号变了。host 脚本硬编码 COM7 失效。
修：所有 host 脚本加 `--port` 参数，GUI 自动检测。

### B2: 颜色"全白"过曝
`brighten 2.5` 把所有>100 的纹理值 clip 到 255 → 颜色去饱和变白色。
修：换成 **保色调 brighten** — 找 max channel，按比例缩放，max clip 到 255 但比例不变。

### B3: 颜色"乱斑"（绿+紫主导）
12K 点 stream 时 UART FIFO 溢出（每帧 60 KB stream，CPU 渲染期间 800ms+ 累积超过 64 byte FIFO），RGB 数据被随机化。
修：在长循环里增加 UART poll 频率（每 128 点一次）。

### B4: cmd.exe TCL 脚本 `[xxx]` 解析为命令替换
`puts "[dl] starting"` 在 TCL 里被当作执行命令 `dl`。改成 `puts "DL: starting"` 就正常了。

### B5: PL batch IP 不更新 model
ARM 写新点云到 DDR，刷 cache，IP 收到 ap_start 但**输出始终是旧 model 渲染的结果**（slot 0 xor 不变）。
猜测：HLS local_model BRAM 持久化跨 ap_start，或者 m_axi/HP/smc 路径有 read cache。
**临时方案**：用 USE_PL=0（CPU 路径）— 19 FPS 但稳定。
**未解决**，标记为后续工作。

### B6: HDMI 双缓冲 park_ptr 写入但 curR 不变
VDMA `XAxiVdma_StartParking` 写 PARK_PTR 寄存器，diag 读回值正确，但 `curR` 状态读永远停在某个固定值。
**未解决**。当前用 circular 模式 + 两个 fbs 都指向 FB_A（等效单缓冲），有撕裂但不影响 LED 板（最终 LED driver 不用 VDMA）。

## 关键决策（写给后续接手的人）

1. **点云 vs 三角面架构**：点云 + 压缩。原因：板子简单（只做投影 + 写 fb），三角面 raster on FPGA 工作量 80-100 hr 不值
2. **CPU 渲染 vs PL IP**：当前 CPU 路径稳定 ~20 FPS。PL batch IP 模式快 6×（125 FPS）但 model-update 有 bug 待修
3. **HDMI 双缓冲撕裂**：known issue，最终 LED 板不依赖 VDMA，可暂留
4. **颜色看起来暗**：glb 纹理本身是 albedo（漫反射率），没 PBR 光照所以偏暗。后续可加 lambertian 或 host 端 pre-bake
5. **UART 是真瓶颈**：115200 baud 限制了大模型流速。WiFi/ETH 是产品阶段必备升级（用 ESP32 桥比直接调 PS ETH 简单）
